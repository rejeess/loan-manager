## Commands

```bash
bun install        # Install dependencies (generates bun.lockb)
bun dev            # Start dev server (Next.js)
bun build          # Production build
bun lint           # ESLint (next/core-web-vitals + next/typescript)
bun typecheck      # tsc --noEmit (run this before committing)

bunx drizzle-kit generate   # Generate a new migration from schema changes
bunx drizzle-kit migrate    # Apply pending migrations to the local DB
bun drizzle/seed.ts         # Seed the local database
```

No test runner is wired yet. When `vitest` and `playwright` are added per the build plan, tests live in `tests/unit/` and `tests/e2e/`. Run unit tests with `bun test` (Bun's built-in runner is vitest-compatible).

## What this app is

A private PWA for two Kerala finance companies — **Jeevana Loans** and **Phenix Money & More** — operated by an owner and clerks. Customers never log in; they only receive WhatsApp messages. The two companies share one codebase and one database, isolated by `company_id` with application-level authorization checks.

## Current state

The app is a PWA shell with mock data (Phase 0 complete). Auth is simulated via `localStorage` in `components/auth/session.ts`; users come from `data/seed.ts`. The next milestone (Phase 1) adds Better Auth, Drizzle migrations, and database-backed data. Do not confuse the mock layer with the intended architecture described below.

## Architecture

**Stack:** Next.js 15 App Router · TypeScript strict · Bun · SQLite (`bun:sqlite`) + Drizzle ORM · Better Auth · Vercel · shadcn/ui + Tailwind

**Routing:** App Router only. Server Components by default. Add `"use client"` only for genuine interactivity (forms, charts, optimistic updates). Never `useEffect` to fetch — use TanStack Query on the client or RSC on the server.

**Planned directory layout** (from `docs/architecture-hld.md`):
```
app/
  (auth)/          # /login, /forgot-password
  (app)/           # authenticated shell — sidebar, company switcher
    dashboard/
    customers/[dcs]/
    loans/[id]/
    payments/
    reports/
    messages/
    settings/
    admin/         # owner-only: roles, audit log
  api/             # Route Handlers (cron, exports, webhooks)
components/
  ui/              # shadcn components
  domain/          # Business display components (RatingBadge, etc.)
  layout/          # AppShell, Sidebar, CompanySwitcher
lib/
  db/              # Drizzle client + query helpers — server only
  auth.ts          # Better Auth instance
  domain/          # Pure business logic: interest, fines, rating, eligibility
  schemas/         # Zod schemas shared between client and server
  audit/
  utils/
drizzle/
  migrations/      # SQL migrations (generated by Drizzle Kit)
  schema.ts        # Drizzle schema definition
  seed.ts          # Seed data
```

**Key invariant:** All business rules (interest, fines, rating, eligibility, recovery) live as pure functions in `lib/domain/`. No database access, no UI imports. Server Actions and cron jobs call these functions, then persist and audit.

## Critical conventions

**Money:** always `bigint` paise in domain and DB. ₹1.00 = 100. Display via `formatINR()` (`lib/utils/currency.ts`). Never use floats or `numeric` for money.

**Time:** store as ISO 8601 text in UTC. Display in `Asia/Kolkata`. Use `date-fns` + `date-fns-tz`.

**TypeScript:** strict mode, no `any`, no `@ts-ignore`, no `as` casts except at parsing boundaries. Prefer `type` for unions, `interface` for object shapes.

**Naming:** files → `kebab-case.tsx`, components → `PascalCase`, DB → `snake_case`, constants → `SCREAMING_SNAKE_CASE`. No barrel files (`index.ts` re-exports) — import directly.

**Server Actions** return `{ success: true; data: T } | { success: false; error: string }` — never throw across the network boundary.

**Database:** SQLite via Bun's built-in `bun:sqlite` driver with Drizzle ORM (`drizzle-orm/bun-sqlite`). All schema changes via migrations in `drizzle/migrations/` — generate with `bunx drizzle-kit generate`, apply with `bunx drizzle-kit migrate`. No RLS — enforce access in Server Actions and middleware. Use `integer` for money (paise), `text` for timestamps (ISO 8601) and UUIDs (generated with `crypto.randomUUID()`), `text` for JSON. Foreign keys always explicit.

**Auth:** owner role requires 2FA. Do not build customer login — customers receive messages only. Do not put service-role credentials in client components.

**Service Worker / PWA:** keep the manifest and service worker installability intact as features are added.

## Roles

- `owner` — access to both Jeevana and Phenix; can see all data, manage users, export
- `clerk` — access to one company; cannot access admin routes

## State layers

- **Server state:** SQLite/Drizzle data through Server Components, Server Actions, TanStack Query
- **Session state:** Better Auth session + user/company membership resolved server-side
- **App UI state:** selected company, language, nav state, form drafts → React context (`CompanyProvider`)
- **Derived business state:** never stored; computed from events or SQLite views/query helpers

Avoid Redux/Zustand until there is a concrete problem that React context + server state cannot solve.
